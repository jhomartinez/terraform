Explicación del Diagrama

Inicialmente colocamos de entrada a Cloud Front debido a que es un servicio para optimizar el uso de sitios/portales web, con este servicio podemos integrar 2 más que nos ayudan al manejo de los DNS y la seguridad con el certificado SSL que nos provee Certificate Manager, asumiendo que es una SPA le di un origen a Cloud front para que use  un bucket en S3 para almacenar nuestro codigo, este último servicio también podemos ubicarlo en varias zonas de disponibilidad si así lo queremos.
Desde nuestro web site podemos consumir apis de otros servicios REST o Websockets haciendo uso de API Gateway, en este caso  podríamos ver la información obteniéndola desde una base de datos a la que nos permiten el acceso (RCC), desde nuestra web podemos conectarnos  haciendo uso de API GATEWAY que a su vez ejecuta una función lambda que tiene embebido codigo java, javascript u otro lenguaje de programación soportado que se conecta con una base de datos externa en donde tenemos acceso haciendo uso de un RDS Proxy que actúa básicamente como un patron singleton para evitar llenar nuestra base de datos final de muchas conexiones.
Por Otro lado tenemos Balanceadores de carga que se conecten a Subdominios que provean aplicaciones o servicios hacia nuestro front, en este caso nos enfocamos a la ejecución del sistema de Facturación, en donde seguramente existen procesos batch, gestión de procesos de notificación, etc que pueden ser pesados sino se integra un balanceador en conjunto con un grupo de autoescalado y 2 o mas zonas de disponibilidad.
Nuestro balanceado de carga de aplicaciones maneja todo el trafico http y https y envía el trafico de forma organizada hacia su grupo objetivo, es decir las Ec2 asociadas disponibles, estas a su vez están dentro de una subnet en conjunto con cluster de amazon RDS ubicados en 2 zonas de disponibilidad y todo dentro un mismo VPC y más globalmente todo dentro de una misma región.
